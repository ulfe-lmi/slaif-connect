# SLAIF Connect Threat Model

This table tracks the practical threats for the relay-only starter. Acceptance
criteria are written as checks future PRs must satisfy before production use.

| Threat | Mitigation | Acceptance criteria |
|---|---|---|
| Relay reroutes to fake SSH server | Extension pins the expected HPC host key or host CA and verifies it before any user authentication prompt. | A fake SSH server behind the relay causes a host-key failure, no password/OTP/private-key prompt is shown, and the session stops. |
| Relay becomes open TCP proxy | Relay token maps server-side to an approved session and HPC alias; relay ignores or rejects client host/port fields. | Requests containing arbitrary `host` or `port` never influence `net.connect`; unknown aliases are rejected. |
| Relay token wrong-scope or replay use | Tokens are scoped (`slaif.launch`, `slaif.relay`, `slaif.jobReport`), short-lived, and consumed on accepted use. | Wrong-scope, expired, and replayed relay tokens are rejected before TCP connection; browser token lifecycle tests prove replay failure. |
| Relay connection resource exhaustion | Relay enforces unauthenticated timeout, auth message size, per-session connection limits, idle timeout, and max connection lifetime. | `npm run test:relay-hardening` covers timeout and oversized auth rejection without Docker. |
| Compromised SLAIF web app requests malicious command | Extension policy owns command templates; web input provides only validated identifiers. | The extension rejects arbitrary command strings and builds only `/opt/slaif/bin/slaif-launch --session ${SESSION_ID}` style commands. |
| Compromised SLAIF web app requests disallowed payload | Signed policy owns `allowedPayloads`, and each host explicitly lists `allowedPayloadIds`; web input may request only a `payloadId`. | Unknown, malformed, or host-disallowed payload IDs are rejected before SSH starts. |
| Malicious payload catalog hides command text | Payload catalog validation rejects command/script/token/credential fields and allows only bounded metadata. | Payload catalog tests reject `command`, `scriptText`, `jobScript`, tokens, passwords, OTPs, and private keys. |
| Remote launcher is replaced with arbitrary shell execution | Launcher is site-controlled code with a documented CLI contract; signed policy points to a fixed launcher path and no descriptor fields can supply scripts or commands. | Launcher tests reject arbitrary command/script flags, validate session IDs, and browser job-reporting E2E exercises the reference launcher in local test mode. |
| Compromised SLAIF web app supplies SSH target fields | Launch message schema allows only alias/session/token; forbidden target fields are rejected. | Messages containing `sshHost`, `sshPort`, `knownHosts`, `command`, or SSH options return `{ok:false}` and do not open a session page. |
| Compromised SLAIF API descriptor redirects SSH | Descriptor may provide relay URL/token only; extension policy owns SSH host, port, known_hosts, HostKeyAlias, and command. | Descriptors containing SSH target fields or command fields fail validation before relay connection or SSH startup. |
| Compromised API serves malicious HPC policy | Extension accepts only signed policy from a trusted SLAIF policy key; descriptor cannot supply policy fields. | Tampered policy, wrong signer, expired policy, and relay-origin mismatch tests fail before SSH starts. |
| Compromised API asks for transcript upload | Descriptor schema rejects transcript/stdout upload fields and arbitrary report URLs; extension reports only safe scheduler metadata. | Descriptors containing `stdoutUploadUrl`, `transcriptUploadUrl`, `jobCommand`, or `schedulerCommand` fail validation and SSH does not start. |
| Job report leaks decrypted terminal data | Reporter builds a minimal payload from validated metadata and never includes raw stdout/stderr/transcript or tokens. | Job reporter tests and browser job-reporting E2E verify forbidden fields are absent from payloads received by the mock API. |
| Unverified real-HPC host key is signed | Pilot collection labels `ssh-keyscan` output as candidate-only; fingerprint verification requires an operator-supplied expected SHA256 fingerprint from an independent source. | No real pilot policy is signed without documented out-of-band verification of the host key or host CA. |
| Pilot tooling leaks real credentials | Pilot stack does not ask for passwords/OTPs in the terminal and does not accept credential environment variables; authentication remains in browser-side SSH. | `npm run test:pilot` needs no real credentials and no pilot script uses `sshpass` or password env vars. |
| Old weaker policy is replayed | Extension stores accepted production policy sequence/fingerprint and rejects lower sequence or same-sequence changed content. | Rollback tests reject lower sequence and same sequence with different fingerprint. |
| Compromised relay logs SSH payloads | Relay logs only session metadata, byte counts, lifecycle events, and errors. | Code review and log tests show no binary payload, terminal text, password prompt, OTP, or private-key material is logged. |
| Relay token theft | Tokens are short-lived, single-use or session-bound, scope-bound, and not placed in URLs. | Missing, malformed, wrong-scope, expired, reused, or wrong-session tokens are rejected before TCP connection. |
| Multi-instance token replay | Production API/relay instances share durable token state with atomic consume semantics. | Redis token-store tests prove concurrent consume allows exactly one success across store instances; deployment config rejects unsafe production memory-token-store use unless explicitly labeled single-instance pilot. |
| Observability leaks secrets or high-cardinality data | Audit redaction removes token/payload-like fields; metrics registry rejects token, session ID, credential, transcript, stdout/stderr, and payload labels. | `npm run test:observability` and browser observability smoke tests verify tokens, SSH payloads, transcripts, credentials, and raw output are absent from audit logs and metrics. |
| Unsafe production runtime config | Deployment config validation rejects insecure production URLs, wildcard origins, missing allowlists, disabled rate limits, disabled audit/metrics, memory audit sinks, and unsafe limits. | `npm run test:deployment` covers unsafe production config rejection and safe summary redaction. |
| Readiness hides missing dependencies | Readiness checks fail when deployment config, token store, rate limiter, relay allowlist, audit sink, metrics registry, signed policy, or trust roots are missing where required. | Health/readiness helper tests prove missing required checks return `ok:false` without leaking secrets. |
| Unknown or changed SSH host key | Production policy requires pinned host keys or a pinned host CA; no automatic acceptance. | Unknown and changed keys fail closed; there is no `StrictHostKeyChecking=no` equivalent in production. |
| Malicious web origin launches extension | Manifest `externally_connectable` is narrow and background code validates `sender.url`. | Messages from unlisted origins receive `{ok:false,error:"origin_not_allowed"}` and no session page opens. |
| Session ID command injection | Session IDs use a strict safe regex before command construction. | Values containing spaces, quotes, semicolons, shell metacharacters, slashes, control characters, or non-ASCII are rejected. |
| Compromised extension build or remote code loading | Extension bundles executable JS/WASM and vendors upstream at build time from pinned source. | Manifest/CSP and build review show no executable remote JS/WASM imports from GitHub, Gitiles, CDN, or SLAIF servers. |
| Relay availability failure | UI surfaces failures clearly; relay closes WSS/TCP cleanly; future retries are explicit. | Dropped, throttled, or reset relay connections terminate the SSH session with a visible error and no credential leakage. |

Residual risk: the relay always sees metadata such as timing, traffic volume,
selected approved alias, and connection success or failure. Treat relay logs as
sensitive operational data.
